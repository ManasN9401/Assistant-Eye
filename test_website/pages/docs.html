Quick start v3.2

Last updated: 14 May 2025 · 18 min read

This guide walks you through creating your first NovaSuite workspace, inviting your team, and shipping your first project — from zero to shipped in under 30 minutes.

Prerequisites: A NovaSuite account (free tier works), Node.js 18+ if using the CLI, and basic familiarity with project management concepts.

1. Installation

NovaSuite is primarily a web application, but the CLI is required for local development integrations, webhook forwarding, and the VS Code extension backend.

Install the CLI

# Via npm (recommended)
npm install -g @novasuite/cli

# Via pip (Python users)
pip install novasuite-cli

# Verify installation
nova --version
# → novasuite-cli 3.2.4
Windows users: Run your terminal as Administrator for the first install. WSL2 is supported but requires the --wsl flag on all subsequent commands.

Authenticate

nova auth login
# Opens browser → confirm in NovaSuite → returns token
# Token stored at ~/.novasuite/credentials.json

# For CI environments, use a service account token:
nova auth login --token $NOVA_SERVICE_TOKEN

2. Create a workspace

A workspace is the top-level container for your organisation. Projects, members, billing, and integrations all live inside a workspace. You can belong to multiple workspaces.

nova workspace create "Acme Corp" --slug acme --region eu-west-1
# Output:
# ✓ Workspace created: acme (ID: ws_8f3kd92)
# ✓ You are the workspace owner
# ✓ Dashboard: https://app.novasuite.io/acme
Tip: Choose your region carefully — data residency is locked at workspace creation and can only be changed on the Enterprise plan. Available regions: us-east-1, eu-west-1, ap-southeast-1.

3. Your first project

Projects organise work around a goal, product, or team. Each project has its own task lists, docs, timelines, and settings. Projects can be public (visible to all workspace members) or private (invite-only).

nova project create "Website Redesign" \
  --workspace acme \
  --template software-development \
  --visibility private

# List available templates:
nova project templates list

Templates available

  • blank — empty project, no predefined statuses
  • software-development — Backlog, In Progress, In Review, Done
  • marketing-campaign — Ideation, Planning, Execution, Reporting
  • bug-tracker — Triage, Confirmed, In Progress, Fixed, Closed
  • okr-planning — Objectives and key results with scoring
  • sprint — Two-week sprints with velocity tracking

4. Creating tasks

Tasks are the core unit of work. Every task has a title, status, assignee, due date, and optional custom fields defined by your project template. Tasks support markdown in descriptions, file attachments up to 100MB, and threaded comments.

# Create a task via CLI
nova task create \
  --project ws_8f3kd92/website-redesign \
  --title "Design new homepage hero" \
  --assignee alice@acme.com \
  --due 2025-06-30 \
  --priority high \
  --status "In Progress"

# Or use the API directly:
curl -X POST https://api.novasuite.io/v1/tasks \
  -H "Authorization: Bearer $NOVA_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "project_id": "proj_4x8w12",
    "title": "Design new homepage hero",
    "assignee_id": "usr_alice",
    "due_date": "2025-06-30",
    "priority": "high",
    "status": "in_progress"
  }'

5. Connect GitHub

The GitHub integration enables two-way sync between NovaSuite tasks and GitHub issues and pull requests. When a PR is opened referencing a task ID (e.g. NOVA-142), the task automatically transitions to "In Review".

# Authorise GitHub integration from the CLI:
nova integration connect github --workspace acme

# Map a GitHub repo to a NovaSuite project:
nova integration github link \
  --repo acme-org/website \
  --project website-redesign \
  --sync-direction bidirectional
Note: Bidirectional sync requires the GitHub app to be installed on your organisation, not just your personal account. The authorising user must be a GitHub org admin. If you're on the Pro plan, only one-directional sync is available (GitHub → NovaSuite).

6. Webhooks

Webhooks allow external services to be notified of events inside NovaSuite. Each workspace can have up to 5 webhooks on Pro, 50 on Business, and unlimited on Enterprise. Events are delivered within 30 seconds, with automatic retries at 1m, 5m, 15m, and 1h intervals.

# Register a webhook
nova webhook create \
  --workspace acme \
  --url https://your-service.example.com/nova-webhook \
  --secret $WEBHOOK_SECRET \
  --events task.created,task.updated,task.completed,comment.created

# Webhook payload example (task.completed):
{
  "event": "task.completed",
  "timestamp": "2025-05-14T10:32:44Z",
  "workspace_id": "ws_8f3kd92",
  "actor": { "id": "usr_alice", "email": "alice@acme.com" },
  "data": {
    "task_id": "task_19xk22",
    "title": "Design new homepage hero",
    "completed_at": "2025-05-14T10:32:44Z",
    "project_id": "proj_4x8w12"
  }
}

7. API rate limits

Rate limits are enforced per API token, not per IP. Exceeding a limit returns a 429 Too Many Requests response with a Retry-After header.

  • Pro: 1,000 requests per day, burst of 60 per minute
  • Business: 50,000 requests per day, burst of 600 per minute
  • Enterprise: Unlimited (fair use policy applies)

Use the X-RateLimit-Remaining and X-RateLimit-Reset headers to manage your usage. The Python SDK handles retries and backoff automatically.

8. Common issues

CLI auth fails silently

If nova auth login opens a browser tab but never returns a success message, your firewall may be blocking the local callback on port 9876. Use the --no-browser flag and paste the authorization code manually:

nova auth login --no-browser
# → Visit: https://app.novasuite.io/cli-auth?code=...
# → Paste the code when prompted:

Webhook deliveries failing

Check the delivery log in Workspace Settings → Integrations → Webhooks → Delivery History. A 401 from your endpoint usually means the X-Nova-Signature header validation is failing. Ensure you're computing the HMAC-SHA256 over the raw request body (not the parsed JSON) using your webhook secret.

Task sync loops with Jira

If you have both the Jira integration and the GitHub integration active, and your Jira project also has a GitHub integration, you may experience sync loops where a status update triggers both systems to update each other. Add nova-sync-skip to the Jira issue labels to prevent NovaSuite from processing events originating from Jira.
